# LightRoute 轻途 视频录制脚本和展示提纲

## 1. 视频目标

用 3-5 分钟让评委理解 LightRoute 轻途的核心价值：它不是让大模型直接编路线，而是把城市微行程规划拆成结构化意图识别、真实 POI 召回、UGC / 天气 / 营业时间补齐、真实路径代价矩阵和确定性路线规划。

## 2. 推荐演示顺序

1. 打开 README，展示项目一句话定位和架构图。
2. 打开 `tools/README.md` 或 `docs/architecture.md`，强调 `poi_search` 和 `route_planning` 是工具。
3. 打开 CLI，输入美食短途案例。
4. 展示路线偏好选择、起点识别、执行过程实时反馈。
5. 展示输出中的路线顺序、预计时间、距离、交通方式、warnings。
6. 快速切换展示 citywalk / 雨天约会 / 下班夜宵等案例说明泛化能力。
7. 总结项目亮点和已知改进方向。

## 3. 3-5 分钟讲解脚本

### 0:00-0:30 开场

屏幕展示：`README.md` 顶部。

讲解词：

> 大家好，这是 LightRoute 轻途，一个城市内短途游和微行程智能规划助手。它解决的是 3 到 6 小时城市活动怎么安排的问题，比如从国贸出发想吃本地特色、下班后按摩再吃夜宵、雨天情侣看展再找小酒馆。传统大模型直接生成路线容易编地点、忽略距离和营业时间，所以 LightRoute 把规划拆成可验证的工具链。

### 0:30-1:10 架构说明

屏幕展示：`docs/architecture.md` 第一张 Mermaid 图。

讲解词：

> 这条链路里，LLM 只负责结构化意图识别。用户输入先进入 CLI，再由 IntentionAgent 输出路线偏好、权重、城市微行程 profile 和活动序列。之后 Orchestrator 调用两个确定性工具：poi_search 负责真实地点召回，route_planning 负责路线规划。最后 plan-trip 只消费 route_options，把结构化结果整理成人类可读的行程，不重新编路线。

### 1:10-2:25 CLI 主案例演示

屏幕展示：终端。

推荐输入：

```text
北京短途游，从国贸出发，6小时，想吃本地特色，不想排队
```

路线偏好选择：

```text
2
```

讲解词：

> 这里我输入一个美食短途需求。系统会先让我选择路线偏好，也可以直接回车让系统自动判断。这个偏好不会变成硬编码路线，而是转换为 food、sightseeing、queue、cost 等权重。接着 CLI 会实时显示处理过程：读取记忆、识别意图、检索真实 POI、优化路线。

屏幕上应展示：

- 路线偏好选择菜单。
- “已收到路线需求”的快速反馈。
- “正在识别你的出行意图 / 正在调度工具和智能体 / 正在检索真实地点 / 正在优化路线”的过程。
- 最终路线标题、顺序、时间、距离、Route Options、注意事项。

推荐展示结果：

```text
北京智能路线规划
出发: 国贸
短时路线安排
Route Options
  - 少排队美食路线: POI A -> POI B -> POI C
    用时约 5-6 小时，排队风险较低，预算约若干元
注意事项
  - 营业时间、排队和天气请出发前确认
```

### 2:25-3:20 核心实现亮点

屏幕展示：`tools/poi_search_tool.py`、`tools/route_planning_tool.py` 或 `docs/technical_solution.md`。

讲解词：

> LightRoute 的核心创新是职责分离。poi_search 根据 activity_sequence 生成 AMap 召回 spec，结合起点、城市、天气、同行人和社交氛围召回 POI，再用 UGC 补齐排队、适合人群和提示。route_planning 是本地确定性工具，不调用 LLM。它会把起点作为虚拟节点加入路径矩阵，至少串联 3 个 POI，并综合时间、距离、换乘、天气、营业状态、POI 质量和偏好匹配进行评分。

### 3:20-4:20 多场景展示

屏幕展示：README 或 `docs/technical_solution.md` 示例表。

讲解词：

> 除了美食路线，系统还能识别更生活化的城市微行程。比如“我在天安门，想进行 3 小时 citywalk”，它会偏向步行、低强度、近距离的公共空间和历史街区；“下雨了，想和女朋友看展再找安静小酒馆”，它会识别雨天、情侣、室内、展览和小酒馆这两个活动槽；“下班按摩再夜宵”，它会强制先 wellness 再 late_night_food，而不是随便找几个关键词相似的地点。

### 4:20-5:00 总结

屏幕展示：`docs/architecture.md` Agent / Tool 关系图。

讲解词：

> 总结一下，LightRoute 轻途的特点是：LLM 做结构化意图，不直接编路线；真实 POI、UGC、天气、营业时间和真实路径代价参与规划；POI 召回和路径规划职责分离；工具链有严格失败和 diagnostics，方便测试、调试和现场解释。后续我们会继续完善真实 UGC 数据源、Web 界面和地图可视化。

## 5. 六组展示案例

### 案例 1：美食短途

输入：

```text
北京短途游，从国贸出发，6小时，想吃本地特色，不想排队
```

展示重点：

- 用户意图：美食优先、低排队、北京短途。
- 活动序列：特色餐饮 -> 轻量文化 / citywalk -> 特色餐饮。
- POI 召回重点：老字号、小吃、本地特色、低排队。
- 路径规划重点：food 和 queue 权重高，起点国贸进入矩阵。
- 为什么合理：满足“吃本地特色”，用低强度节点连接路线。
- warning：热门餐厅排队和营业时间需确认。

### 案例 2：拍照打卡

输入：

```text
北京短途游，从国贸出发，6小时，想多拍照打卡，少排队
```

展示重点：

- 用户意图：打卡拍照路线。
- 活动序列：地标 / 文化街区 -> 拍照点 -> 轻餐饮。
- POI 召回重点：地标、热门景点、夜景、网红拍照。
- 路径规划重点：sightseeing 和 queue 权重高。
- 为什么合理：强化视觉体验，餐饮不喧宾夺主。
- warning：户外拍照受天气影响。

### 案例 3：citywalk

输入：

```text
我在天安门，想要进行3小时的citywalk，请为我推荐一条轻松的路线
```

展示重点：

- 用户意图：低强度步行。
- 活动序列：地标外观 -> 历史街区 -> 休息点。
- POI 召回重点：天安门周边、胡同、公共空间、公园。
- 路径规划重点：walking / multimodal_low_friction，控制距离和换乘。
- 为什么合理：3 小时不适合远距离折返。
- warning：部分点位仅建议外观打卡。

### 案例 4：下班放松夜宵

输入：

```text
北京，我一下班想去按摩放松，然后吃个夜宵，大概3小时
```

展示重点：

- 用户意图：下班后放松。
- 活动序列：wellness -> late_night_food。
- POI 召回重点：按摩、足疗、SPA、夜宵。
- 路径规划重点：先按摩后夜宵，晚间营业。
- 为什么合理：符合用户明确顺序和低能量状态。
- warning：夜间营业信息需确认。

### 案例 5：情侣雨天约会

输入：

```text
下雨了，想和女朋友在北京约会，看看展览，再找个安静小酒馆，4小时
```

展示重点：

- 用户意图：雨天情侣室内约会。
- 活动序列：museum_exhibition -> drinks。
- POI 召回重点：展览、美术馆、博物馆、安静酒馆。
- 路径规划重点：雨天室内优先，conversation friendly。
- 为什么合理：减少户外暴露，活动顺序符合约会节奏。
- warning：展览预约和酒馆营业时间。

### 案例 6：闺蜜美甲小酒

输入：

```text
今天下午无事可做，和闺蜜想去做指甲和点小酒，大概5小时行程
```

展示重点：

- 用户意图：闺蜜下午休闲。
- 活动序列：beauty -> drinks -> 可选轻餐 / 拍照。
- POI 召回重点：美甲、美容、清吧、小酒、适合聊天。
- 路径规划重点：过滤培训、诊所等错误 POI。
- 为什么合理：先美甲后小酌，符合时间和社交氛围。
- warning：美甲预约和下午酒馆营业。

## 6. 项目亮点总结

- LLM 不直接编路线，只做结构化意图识别。
- route_planning 是本地确定性工具，便于测试和复现。
- POI 召回、路径规划、行程展示职责分离。
- 结合真实 POI、UGC、天气、营业时间和路径代价矩阵。
- 支持多种城市微行程场景。
- 严格失败和 diagnostics 让系统更容易解释。
